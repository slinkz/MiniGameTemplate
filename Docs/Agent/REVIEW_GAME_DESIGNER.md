# 游戏设计师视角评审：弹幕系统 & VFX 系统需求盲区

> 评审日期：2026-04-11 | 评审角色：游戏设计师（以真实弹幕/STG 项目开发流程为基准）

---

## 一、评审方法论

从一个真实竖屏弹幕手游（如 弹幕与飞行：Phoenix 2、ArchHero 等）的**完整开发需求**出发，逐项比对当前系统能力，标注覆盖/缺失/部分覆盖。不局限于弹幕射击游戏——弹幕系统在塔防、RPG Boss 战、动作游戏中同样有广泛应用。

---

## 二、需求盲区清单

### 🔴 P0 — 根本缺陷（不解决就无法支撑真实项目）

| ID | 需求 | 现状 | 问题描述 |
|----|------|------|----------|
| **GD-001** | 多贴图共存 | ❌ 缺失 | 真实项目中弹丸类型数十种（圆弹、米粒、蝴蝶、符文、Boss 专属弹等），不可能全部塞进一张图集。当前 `DanmakuRenderConfig` 只有一个 `BulletAtlas`，所有弹丸共享同一张贴图。VFX 系统同理。**这是最大的架构硬伤。** |
| **GD-002** | 弹丸视觉多样性 | ❌ 缺失 | 真实弹幕游戏的弹丸需要：缩放动画（出生时从小变大）、Alpha 渐变（死亡前淡出）、颜色渐变（生命周期变色）、闪烁效果。当前 `BulletTypeSO` 只有静态 `Tint` 和 `Size`，无任何动画曲线。 |
| **GD-003** | VFX 与弹幕系统的耦合方式 | ❌ 缺失 | 弹丸碰撞产生爆炸、擦弹产生特效、Boss 释放技能附带特效光环——这些是弹幕游戏最基本的表现需求。当前 DanmakuSystem 和 VFXSystem 是完全独立的两个 MonoBehaviour，没有任何事件通道或回调接口。`DanmakuSystem` 虽有 `_hitVfxSystem` 字段但只支持单一特效类型。 |
| **GD-004** | 喷雾系统无可视化 | ❌ 缺失 | `SprayUpdater` 只做逻辑更新，`SprayRenderer` 完全不存在。喷雾/锥形 AOE 在弹幕游戏中极为常见（Boss 火焰吐息、毒雾区域），不可视化等于不存在。 |

### 🟡 P1 — 重要缺失（不解决会严重限制设计空间）

| ID | 需求 | 现状 | 问题描述 |
|----|------|------|----------|
| **GD-005** | 弹丸运动多样性 | ⚠️ 部分覆盖 | 有直线、追踪、延迟变速。缺少：**正弦波运动**（蛇形弹）、**螺旋运动**、**贝塞尔曲线路径**、**弹丸悬停后二次加速**。真实弹幕游戏中这些是标配。 |
| **GD-006** | 多阵营系统 | ⚠️ 部分覆盖 | 有 `BulletFaction.Player/Enemy`，但只有两个阵营。PvP 弹幕、三方混战（玩家 vs 敌人 vs 环境）需要更灵活的阵营/碰撞过滤。 |
| **GD-007** | 弹幕演出编排 | ⚠️ 部分覆盖 | `PatternGroupSO` + `PatternScheduler` 能做基本的延迟/重复/旋转编排。缺少：**可视化编排工具**、**运行时预览**、**与 Boss 状态机联动的条件触发**（如 Boss 血量低于 50% 切换弹幕模式）。 |
| **GD-008** | 屏幕清除（Bomb/清屏技能）| ❌ 缺失 | 弹幕游戏标配的「清屏炸弹」需要：将存活弹丸转化为得分道具/特效、播放全屏闪白、短暂无敌。当前 `FreeAll()` 只是静默清除。 |
| **GD-009** | 弹丸与弹丸碰撞 | ❌ 缺失 | 玩家子弹打掉敌人弹丸（消弹）是常见玩法。当前碰撞只有 弹丸vs目标、弹丸vs障碍物，没有弹丸vs弹丸。 |
| **GD-010** | 道具/收集物 | ❌ 缺失 | 敌人死亡掉落能量/金币/增益道具，视觉上也是「从空中飘落的带碰撞物体」，与弹丸共享渲染/碰撞管线。当前系统完全没考虑非伤害性的弹丸实体。 |
| **GD-011** | 弹幕预警线 | ❌ 缺失 | 激光发射前显示红色预警线、大型弹幕释放前屏幕边缘闪烁提示。这是弹幕游戏的基本 UX 设计。当前激光虽有 Charging 阶段但无视觉预警。 |
| **GD-012** | Z 轴排序 / 渲染层级控制 | ❌ 缺失 | 弹丸需要在玩家角色下方还是上方？不同类型弹丸需要不同的渲染顺序。当前所有弹丸统一在一个 Mesh 中，没有 Z 排序或 sortingOrder 控制。 |

### 🟢 P2 — 增强需求（优化用户体验和设计空间）

| ID | 需求 | 现状 | 问题描述 |
|----|------|------|----------|
| **GD-013** | 子弹时间（慢动作）| ✅ 已有 | `DanmakuTimeScaleSO` 已实现，但 VFXSystem 不受其影响——慢动作时弹丸慢了但爆炸特效还是原速，视觉不一致。 |
| **GD-014** | 相机震动 | ❌ 缺失 | 大型弹幕/爆炸需要相机震动反馈。当前框架无 CameraShake 模块。 |
| **GD-015** | 弹丸音效分层 | ⚠️ 部分覆盖 | `BulletTypeSO` 有 `BounceSFX`/`PierceSFX`，但缺少：发射音效、飞行循环音效（激光嗡嗡声）、碰撞命中音效分类（轻击/重击）。更重要的是缺少 **音效优先级/预算** 管理——100 颗子弹同时命中会产生 100 个音效请求。 |
| **GD-016** | 弹丸粒子拖尾 | ⚠️ 部分覆盖 | 有 Ghost 残影和 TrailPool 曲线拖尾。缺少：粒子拖尾（每帧在弹丸位置喷射小粒子），这在视觉上更丰富。 |
| **GD-017** | 统计与调试 | ⚠️ 部分覆盖 | `DanmakuDebugHUD` 有基础 OnGUI 信息。缺少：碰撞可视化（显示碰撞半径）、弹幕模式预览（编辑器中不运行游戏就能看到弹幕轨迹）、性能 Profiler 标记（`ProfilerMarker`）。 |
| **GD-018** | 配置热重载 | ❌ 缺失 | 策划调整 SO 参数后需要重启游戏才能看到效果。理想情况是运行时修改 SO 即时生效（ScriptableObject 的 `OnValidate` + 运行时刷新）。 |
| **GD-019** | 容量可配置 | ⚠️ 部分覆盖 | `DanmakuWorldConfig` 的 MaxBullets/MaxLasers/MaxSprays 可配置，但 ObstaclePool(64)、TrailPool(64)、PatternScheduler(64)、SpawnerDriver(8)、DamageNumberSystem(128)、TargetRegistry(16)、AttachSourceRegistry(24) 全部硬编码。 |
| **GD-020** | VFX 类型丰富度 | ❌ 缺失 | 当前 VFX 系统只支持 Sprite Sheet 帧动画。缺少：持续性特效（光环/buff 环绕/能量场）、跟随特效（附着在弹丸或角色身上移动的特效）。 |

## 三、真实项目开发流程对照

### 3.1 策划配置工作流

| 步骤 | 理想流程 | 当前流程 | 差距 |
|------|----------|----------|------|
| 1. 创建新弹丸类型 | 选图集→选子图→配参数→预览 | 手动填 UV Rect 四个浮点数 | 缺少图集选择器/子图预览 Editor 工具 |
| 2. 编排弹幕模式 | 可视化时间轴→节点连线→实时预览 | 手工创建多个 SO 并嵌套引用 | 缺少编排编辑器 |
| 3. 调试弹幕效果 | 运行时 Inspector 拖动参数立即见效 | 修改 SO → 重启游戏 | 缺少热重载 |
| 4. 性能预算 | 设定帧预算→自动降级 | 无预算管理 | 完全缺失 |

### 3.2 程序员集成工作流

| 步骤 | 理想流程 | 当前流程 | 差距 |
|------|----------|----------|------|
| 1. 接入弹幕到新场景 | 拖入预制件→配 SO→完成 | 拖入 DanmakuSystem→手动配 6+ 个 SO 引用 | 初始化配置过多 |
| 2. 碰撞回调 | 注册监听器→收到碰撞事件→处理逻辑 | 修改 CollisionSolver 代码 | 缺少事件化碰撞回调 |
| 3. 添加新弹丸运动类型 | 实现接口→注册→SO 选择 | 修改 BulletMover 硬编码 | 运动逻辑不可扩展 |
| 4. 跨系统联动 | 弹幕→VFX→音效→相机 全自动 | 手动在多个系统间接线 | 缺少统一事件总线 |

## 五、优先级建议（从游戏设计角度）

### 第一批（不解决就不能开发真实关卡）

1. **GD-001 多贴图**：决定渲染架构的基础
2. **GD-002 弹丸视觉动画**：决定视觉表现力的天花板
3. **GD-003 弹幕×VFX 联动**：决定打击感和视觉反馈
4. **GD-004 喷雾可视化**：已有逻辑代码但看不见

### 第二批（开始做第二个关卡/Boss 时必须有）

5. **GD-005 运动多样性**：没有蛇形弹、螺旋弹，弹幕游戏就是半成品
6. **GD-008 清屏炸弹**：核心游戏机制
7. **GD-009 消弹**：核心游戏机制
8. **GD-010 道具系统**：核心游戏循环
9. **GD-011 预警线**：基本 UX

### 第三批（产品打磨阶段）

10. **GD-007 编排工具**：提升策划效率
11. **GD-012 渲染层级**：视觉精修
12. **GD-013~020 所有 P2**：体验优化

## 六、对架构师的输入

以上 20 项需求盲区将作为**软件架构师评审的需求基准**。架构评审应回答：

1. 当前架构能否通过迭代修正支撑上述需求？还是需要重新架构？
2. 如果重新架构，核心的架构变更点在哪里？
3. 哪些需求可以通过扩展点（接口/回调）预留，不必立即实现？
