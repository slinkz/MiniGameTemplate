## 一、执行方式

本轮采用“Unity 架构师先挑问题 → 软件架构师逐条答辩 → 项目经理判定是否继续”的节奏推进。

### 回合 1

#### Unity 架构师新增问题

**UQ-001：OnValidate / 域重载 / 热重载生命周期边界未定**
- 现有文档已规定运行时禁止热路径重建，但没有把 `OnValidate`、域重载、资源重导入、禁用 Domain Reload 等 Unity 编辑器生命周期边界写死。
- 风险：若在 `OnValidate` 中直接做 Registry 重建、Batch 预热、Mesh/Material 重建，会导致编辑器反复重建、Play 脏状态、静态缓存残留、Prefab Stage 与主场景不一致等问题。

**UQ-002：统一资源描述的版本化序列化迁移策略未定**
- 文档已经统一了资源描述语义，也要求提供迁移器，但没有把 `SchemaVersion`、迁移顺序、幂等性、Prefab/Scene 实例覆盖值处理策略正式写成制度。
- 风险：后续字段重组、嵌套值对象化后，会再次踩 Unity 序列化迁移坑。

**UQ-003：资源变更 → Registry 刷新 → Batch 预热 的编辑器工作流未定**
- 文档分别规定了资源描述、Registry 重建时机、Batch 预热原则，但没有把三者串成一个可执行的编辑器工作流。
- 风险：策划改 Inspector 后，容易出现“看起来改了但不生效”“Play 一次才生效”“旧桶和旧缓存没刷新”的状态错乱。

#### 软件架构师答辩结论

软件架构师判断：

- 这 **3 个问题都不是新的架构级未决项**；
- 它们属于 **执行契约缺口**，即上位架构方向已经定了，但 Unity 编辑器侧的落地规则还没写成正式条款；
- 因此不需要推翻现有方案，只需要把约束回写成正式 ADR / 计划条款。

#### 软件架构师给出的新增正式条款候选

### SA-ADD-001 ~ SA-ADD-005：OnValidate / 热重载边界
1. `OnValidate` 只允许做数据规范化、兼容修正、dirty 标记、告警输出
2. `OnValidate` 禁止直接触发运行时对象创建、Registry 全量重建、Batch 预热、跨系统刷新
3. 编辑器事件（OnValidate / 域重载 / 资源导入 / UndoRedo / Inspector 改值）统一只进入 Dirty 状态
4. Play Mode 下只允许记录 dirty / queued refresh，不允许在热路径直接重建
5. 必须兼容开启和关闭 Domain Reload 两种模式，静态缓存不能依赖域重载自动清空

### SA-ADD-006 ~ SA-ADD-011：统一资源描述版本化迁移
6. 统一资源描述必须显式带 `SchemaVersion`
7. 迁移必须按版本链路顺序单向推进，且每一步幂等
8. 迁移器必须支持 `dry-run` / `apply` / `report`
9. 迁移覆盖范围必须包含 SO / prefab / scene / variant / 嵌套实例覆盖值
10. Atlas 结果不得成为唯一真相，源事实仍是版本化资源描述
11. 旧字段兼容读取只作为过渡窗口，不能长期双轨共存

### SA-ADD-012 ~ SA-ADD-018：编辑器刷新工作流
12. 固定刷新主链路：资源变更检测 → Registry 标脏 → Registry 重建 → 预热集合重算 → Batch 预热 → 报告输出
13. Dirty 必须按影响域最小化，至少区分 Bullet / VFX / 共享描述缓存 / Atlas 映射缓存
14. 刷新顺序固定，任一步失败必须中止后续步骤
15. 自动刷新与手动刷新并存；大范围变更时自动退化为“仅标脏 + 提示手动刷新”
16. Batch 预热输入只能来自当前有效 Registry 快照
17. 刷新流程必须输出可审计的可观测信息（变更资源数、桶数、失败项、未知项、耗时）
18. 进入 Play Mode 不得隐式补做完整编辑器刷新；Dirty 未清时必须显式告警

## 三、最终结论

### 1. 这轮 PK 的产出是什么
不是推翻方案，而是把之前“默认大家会这么做”的 Unity 编辑器侧规则，升级成必须回写的正式契约。

### 2. 当前方案还能不能开工
**能。**

但要把结论说完整：

- **架构级未决项：无**
- **执行契约缺口：有 3 组**
  1. OnValidate / 域重载 / 热重载边界
  2. 统一资源描述的版本化迁移制度
  3. 资源变更 → Registry 刷新 → Batch 预热 的编辑器工作流

### 3. 现在最合理的动作
不是再开新一轮抽象讨论，而是：

1. 把 ADR-023 ~ ADR-025 正式补进决策记录
2. 把 REFACTOR_PLAN / REVIEW_SOFTWARE_ARCHITECT 同步更新
3. 然后按“架构已定、执行契约补齐”的状态进入 Phase 0
